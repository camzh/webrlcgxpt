# 架构设计规则

## 设计原则

### 先查后建

做任何架构决策前，先完整理解项目现有结构：

1. 使用 Glob 列出项目目录树，确认现有模块、组件、工具函数的分布。
2. 使用 Grep 搜索相似功能的已有实现，确认是否可复用或扩展。
3. 阅读已有的架构文档、README 或相关规则文件。

只有当确认现有代码无法满足需求时，才新建模块或引入新依赖。新建时在方案中说明：为什么不能复用、为什么选择这个方案。

### 为任务场景选架构，不为架构选任务

小程序项目的架构选择应基于实际场景，而非个人偏好或流行趋势：

- 简单页面（纯展示、无复杂交互）：原生 Page/Component 即可，不引入状态管理。
- 中等复杂度（多组件共享状态、表单交互）：考虑全局状态管理或事件总线。
- 高复杂度（多页面联动、复杂数据流）：引入成熟的状态管理方案，但仅在必要时。

避免在项目初期过度设计。架构应随需求复杂度增长而演进，而非预先铺设。

### 模块边界清晰

每个模块应满足：

- 单一职责：一个模块只做一类事。
- 明确的输入输出：接口契约清晰，不依赖其他模块的内部实现。
- 可独立测试：不启动整个应用即可验证模块行为。
- 可替换：替换一个模块的实现不影响其他模块。

### 依赖方向单向

- 上层可依赖下层，下层不可依赖上层。
- 业务逻辑层不直接依赖 UI 框架的具体 API。
- 工具函数不依赖业务模块。

## 技术选型决策框架

引入任何新依赖（npm 包、第三方库）前，过以下检查清单：

1. **必要性** — 项目现有代码能否实现？能否用少量代码替代？
2. **成熟度** — 是否有活跃维护？Star 数、Issue 响应速度、最近更新时间。
3. **体积** — 对小程序包体积的影响是否可接受？（小程序有 2MB 主包限制）
4. **兼容性** — 是否与项目现有技术栈（框架版本、构建工具）兼容？
5. **许可证** — 是否与项目许可证兼容？

对每个决策点，记录选择理由（一条注释即可），便于后续维护者理解。

## 目录结构规范

小程序项目建议遵循以下结构，可根据实际场景调整：

```
src/
├── pages/           # 页面（按业务模块分组）
├── components/      # 公共组件
├── services/        # API 请求、数据处理
├── stores/          # 状态管理
├── utils/           # 纯工具函数
├── constants/       # 常量、枚举
├── styles/          # 全局样式、主题变量
└── app.ts           # 入口
```

规则：
- `utils/` 中的文件不得依赖 `services/`、`stores/`、`components/`。
- `components/` 中的公共组件不得直接引用页面路径。
- `services/` 中的文件不得引入 UI 组件。

## 禁止事项

- 禁止为了"以后可能用到"而预先设计抽象层。
- 禁止在单个文件中混合 UI 逻辑、业务逻辑和数据请求。
- 禁止循环依赖。如果发现 A 依赖 B 同时 B 依赖 A，立即重构。
- 禁止直接操作其他模块的内部状态，必须通过模块暴露的接口。

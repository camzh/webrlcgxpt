# 效率与成本优化规则

## 核心原则

以最小的 token 消耗达到交付标准。每个操作都要问：有没有更省的方式达到同样效果？

## 文件操作效率

### 查找文件

- 优先 Glob（模式匹配），不遍历目录。
- 优先 Grep（内容搜索），不打开文件逐行查看。
- 知道文件名时直接 Glob，知道内容关键词时直接 Grep，不要先用 LS 再用 Read。

### 读取文件

- 优先按需读取（指定行范围），不全文读取。
- 需要了解文件结构时，先读前 50 行判断内容分布，再决定是否读更多。
- 不要为了"全面了解"而通读整个项目。按任务需要，读相关文件即可。

### 编写文件

- 优先编辑现有文件（SearchReplace），不重写整个文件。
- 新增内容超过 100 行时，先写大纲，再分块填充。
- 避免在输出中展示完整文件内容。用差异描述或关键代码段替代。

## 对话效率

### 任务粒度

- 单个对话聚焦一个主题。不要在同一个对话中切换多个不相关的任务。
- 任务完成后，主动总结并询问是否继续下一个任务，而非等待用户指令。
- 如果当前对话上下文已超过 5 轮且任务仍复杂，建议用户开启新对话继续。

### 输出简洁

- 回答直接给出结论，再展开解释。不要用大段铺垫。
- 代码输出只展示关键变更和上下文，不输出完整文件（除非用户明确要求）。
- 解释性文字控制在一屏以内。超过时用折叠或分段。

### 避免重复

- 不在同一个对话中重复解释已讨论过的概念。
- 如果用户多次问同类问题，提取为规则或文档，后续引用即可。

## 工具调用效率

### 并行化

- 多个独立的工具调用可以并行发出（如同时读取多个文件、同时搜索多个关键词）。
- 文件读取和网络搜索可以并行进行。
- 但并行数量不超过 5 个，避免资源竞争。

### 减少冗余调用

- 搜索前先想清楚关键词，一次精准搜索优于多次模糊搜索。
- 读取文件前确认该文件是否已在当前对话中读过。
- 不重复搜索已确认不存在的 API 或文件。

## 交付最小化

### MVP 优先

- 先交付核心功能的最小可用版本，再按需完善。
- 不主动添加用户未要求的功能、"锦上添花"的优化或"以后可能用到"的扩展。
- 如果用户需求模糊，先交付最可能的解释版本，而不是为了覆盖所有可能性而过度实现。

### 文件产出控制

- 不主动创建 README、文档文件（除非用户明确要求）。
- 不创建临时脚本后留在工作区。中间文件放在临时目录，最终只保留交付物。
- 不创建空的占位文件或目录。

### 代码产出控制

- 不写可以用一行注释替代的多行注释。
- 不为单一用途创建抽象层。
- 不引入只为"语义化"而存在的中间变量。

## 连续任务的上下文管理

如果你将在多个连续对话中推进同一个大任务：

1. 每个对话结束时，输出一段简短的状态摘要，供下一个对话的 AI 快速接手。
2. 状态摘要包含：已完成什么、当前进度、下一步做什么、需要关注的风险点。
3. 将关键决策记录在项目规则或 AGENTS.md 中，确保跨对话一致。

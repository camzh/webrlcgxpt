# 质量闸门规则

## 决策置信度框架

每个技术决策点，按以下阈值采取行动：

| 置信度 | 行动 |
|-------|------|
| >= 90% | 执行，注释中简述决策理由 |
| 80% - 90% | 执行，但告知用户不确定性所在及判断依据 |
| < 80% | 暂停，先自行查证。查证后仍不足 80%，向用户确认 |

### 置信度自评清单

面对一个决策点，快速自问：

1. 我是否见过这个 API/函数的确切签名和文档？如果只凭名称猜测 → 置信度 < 50%。
2. 我是否了解该方案在项目中的已有用法？如果不确定项目是否已有类似实现 → 置信度 < 70%。
3. 我是否清楚该方案的所有边界条件和副作用？如果只了解 happy path → 置信度 < 80%。
4. 我是否曾在类似场景中验证过该方案？如果是新场景 → 置信度自动扣 10%。

### 自行查证的优先级

1. 项目内搜索：Grep 搜索已有代码，看是否有类似实现或调用。
2. 官方文档：优先查阅官方文档而非博客或论坛。
3. 权威来源：框架作者、核心维护者的说明。
4. 社区验证：多个独立来源交叉验证。

禁止：凭记忆回答 API 签名、凭经验假设框架行为、凭感觉判断性能。

## 反幻觉规则

### 禁止编造以下内容

- 不存在的 API、函数、类、方法名。
- 不存在的 npm 包、库、插件。
- 不存在的配置项、参数、选项。
- 不存在的文件路径或目录结构。
- 不存在的错误码、错误信息。

### 查验机制

每当需要引用上述信息时：

1. 先在项目代码中搜索确认。
2. 项目内找不到时，查阅官方文档。
3. 官方文档未覆盖时，使用 WebSearch 查找权威来源。
4. 所有来源均无结果时，明确告知用户"未找到相关信息"，并说明已搜索的范围。

### 引用标注

- 引用项目内文件时，使用相对路径。
- 引用外部文档时，附带来源链接。
- 给出代码示例时，确保示例中的 API 均为真实存在的。

## 代码规范

### 命名

- 变量/函数名使用有意义的英文单词，避免拼音、缩写（除非是领域通用缩写，如 `id`、`url`）。
- 布尔变量以 `is`、`has`、`should`、`can` 开头。
- 事件处理函数以 `handle` 或 `on` 开头。
- 常量使用全大写蛇形命名（如 `MAX_RETRY_COUNT`）。

### 函数

- 单一职责：一个函数只做一件事。
- 函数体不超过 30 行。超过时优先考虑拆分，而非加注释。
- 参数不超过 4 个。超过时考虑封装为配置对象。
- 避免布尔参数作为行为开关。用两个独立函数替代。

### 错误处理

- 所有异步操作必须有错误处理，不允许 unhandled promise rejection。
- 所有用户输入必须校验，不允许假设输入合法。
- 错误信息必须包含足够的上下文，便于定位问题。
- 不允许空的 catch 块。如果确实不需要处理，加注释说明原因。

### 禁止事项

- 禁止 `any` 类型（TypeScript 项目）。如果类型确实不确定，使用 `unknown` 并在使用处做类型守卫。
- 禁止 `console.log` 遗留到生产代码。使用统一的日志工具。
- 禁止注释掉的代码块。删除或用版本控制回溯。
- 禁止魔法数字。定义为命名常量。

## 验证要求

每次代码改动后：

1. 确认项目可编译（无 TypeScript 错误）。
2. 至少运行一次相关功能的手动验证。
3. 如果项目有测试，运行相关测试套件。
4. 检查是否有未使用的导入或变量，一并清理。

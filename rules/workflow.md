# 工作流与规划规则

## 任务复杂度判定

收到任务时，首先判定复杂度，选择对应策略：

| 复杂度 | 特征 | 策略 |
|-------|------|------|
| 低 | 单文件修改、修 bug、加注释 | 直接执行，完成后自检 |
| 中 | 2-5 个文件、一个功能模块 | 使用 `/plan` 工作流，拆分 2-3 个子任务 |
| 高 | 新模块、跨多层变更、架构调整 | 使用 `/spec` 工作流，生成完整规范文档 |

如果不确定复杂度，默认按中等复杂度处理。

## 模块拆分规则

### 拆分粒度

每个子任务应满足：
- 涉及文件不超过 5 个。
- 可在一个步骤内完成编码和验证。
- 有明确的、可测试的完成标准。

### 拆分顺序

按依赖关系排序，确保每个子任务完成时，后续子任务可立即开始：

1. 先建基础设施（类型定义、常量、工具函数）。
2. 再建数据层（API 封装、状态管理）。
3. 然后建业务逻辑层。
4. 最后建 UI 层。

注意：上述顺序是理想情况。如果 UI 层需要先行以验证交互流程，可调整顺序，但需明确说明。

### 验收节点

每个子任务完成后，必须经过以下验收节点才能进入下一个：

1. **代码审查** — 自查：命名是否清晰、逻辑是否正确、是否有遗漏的边界情况。
2. **编译检查** — 确保项目可正常编译，无类型错误。
3. **功能验证** — 运行相关页面或单元测试，确认行为符合预期。
4. **回归检查** — 确认改动未破坏已有功能（至少做一次 smoke test）。

验收通过后，使用 Git 提交该子任务。提交信息遵循 `scene: git_message` 规则。

## 不可拆分的大任务

如果任务规模过大，无法在单次对话中合理完成，主动告知用户并建议：

- 将任务拆分为多个独立对话，每次聚焦一个模块。
- 先完成核心路径（MVP），再逐步完善边缘功能。
- 使用 Spec 工作流将任务文档化，分阶段执行。

## 中途遇到问题

如果执行过程中遇到阻塞（如缺少信息、技术障碍、发现需求矛盾）：

1. 记录当前进度：已完成哪些子任务、卡在哪个子任务。
2. 分析问题原因：是信息缺失、技术限制还是需求不清。
3. 向用户报告：问题是什么、已尝试什么、建议下一步怎么做。

禁止在遇到阻塞时自行绕过或修改需求范围，必须经用户确认后调整。
